1.网络结构模式

C/S结构

服务器-客户机结构，服务器负责数据管理，客户机完成与用户的交互任务，应用程序分为两部分，服务器部分和客户机部分，服务器部分执行后台服务，用于多个用户的信息共享等功能，客户机部分为用户所专有，执行前台功能

优点在于可以充分发挥客户端PC的处理能力，许多工作可以在客户端处理后再交给服务器，客户端响应速度快，安全性高

缺点在于客户端需要安装专用的客户端软件，维护与升级成本高，不能够跨平台

B/S结构

浏览器-服务器模式，web浏览器作为客户端，统一了客户端，将系统功能实现额核心部分集中到服务器上，简化了系统的开发、维护和使用

优点在于节约成本、维护方便、分布性强、开发简单

缺点在于通信开销大，安全性低，系统个性特点降低，响应速度慢

2.MAC地址

网卡是使计算机能够在计算机网络上进行通讯的计算机硬件，又称为网络适配器。每一个网卡都拥有一个被成为MAC地址的独一无二的48位串行号

MAC地址又称为局域网地址、以太网地址、物理地址或硬件地址，MAC地址在世界上是唯一的

3.IP地址

IP协议是为了计算机网络能相互连接进行通信而设计的协议，IP协议将各种计算机网络中不同的数据帧统一转换成IP数据报格式

IP地址是指互联网协议地址，是IP协议提供的一种统一的地址格式，它为互联网上每一个网络、每一个主机分配一个逻辑地址，以屏蔽物理地址的差异。

IP地址是32位的，通常被分割为4个八位二进制数(四个号码段)

  • A类IP地址是指IP地址的四段号码中，第一段号码为网络号码，剩下的三段号码为本地计算机的号码
  • B类IP地址是指IP地址的四段号码中，前两段为网络号码，后两段为本地计算机的号码
  • C类IP地址是指IP地址的四段号码中，前三段为网络号码，后一段为本地计算机的号码
  • 子网掩码是通过逻辑运算，将IP地址划分为网络标识与主机标识的32位的二进制数，只有网络表示相同的两台主机在无路由器的情况下才能相互通信

注意，网络地址的最高位必须是0；

4.端口

端口是设备与外界通讯交流的出口。可分为虚拟端口和物理端口，虚拟端口指计算机内部或交换机路由器内的端口，不可见，是特质TCP/IP协议中的端口。
端口是通过端口号来标记的，端口号只有整数(0-65535)，不同主机间的进程要实现通讯，就是通过端口号来识别的。一个应用程序可以有多个端口。

5.网络模型

OSI七层参考模型

至下而上：

  • 物理层：定义物理设备标准，比如网线的接口类型、各种传输介质的传输速率，该层的数据叫做比特
  • 数据链路层：建立逻辑连接，进行硬件地址寻址、差错校验等功能。将比特组合成字节进而组合成帧，用MAC地址访问介质
  • 网络层：进行逻辑地址寻址，在位于不同地理位置的网络中的两个主机系统之间提供连接和路径选择(IP数据报)
  • 传输层：定义了一些传输数据的协议和端口号，如TCP(传输控制协议，传输效率低，可靠性高)，UDP(用户数据报协议，可靠性不高，传输效率高)，这一层的数据叫段
  • 会话层：通过传输层定义的端口建立数据传输通路，在系统之间发起会话或接收会话
  • 表示层：数据的表示，转换成人能够接收的信息
  • 应用层：为应用程序提供服务

TCP/IP四层模型

  • 数据链路层(ARP协议,RARP协议)
  • 网络层(IP协议,ICMP协议)
  • 传输层(TCP/UDP协议)
  • 应用层

6.协议

网络协议是通信计算机双方必须共同遵守的一组约定，只有遵循某个约定，计算机之间才能互相通信交流，其三要素是：语法、语义、时序

  • 应用层常见的协议有：FTP文件传输协议，HTTP超文本传输协议，NFS网络文件系统
  • 传输层常见的协议有：TCP传输控制协议，UDP用户数据报协议
  • 网络层常见的协议有：IP因特网互联协议，ICMP，ICMP
  • 网络接口层常见的协议有：ARP地址解析协议(通过IP地址得到MAC地址)，RARP(通过MAC地址获得IP地址)

上层协议通过封装实现下层协议的服务。应用程序数据发往物理网络之前，将沿着协议栈从上往下依次传递，每层协议都在上层数据的基础上加上自己的头部信息。

当数据帧达到目标主机时，将沿着协议自下而上依次传递。各层协议处理数据帧中本层负责的头部数据，将最终的处理后的帧提交给目标应用程序，该过程称为分用

7.Socket套接字

Socket是由IP地址和端口结合的，提供向应用层进程传送数据报的机制，是对于网络中不同主机上应用进程之间进行双向通信的接口，在Linux系统下，表示为进程间网络通信的一种特殊文件类型，本质为内核借助缓冲区形成的伪文件

字节序

字节序就是指大于一个字节类型的数据在内存中的存放顺序，字节序分为大端字节序和小端字节序。大端字节序值一个整数的高位字节存储在内存的低地址处，低位字节存储在内存的高地址处；小端字节序指整数的高位字节存储在高地址处，低位字节存储在低地址处。
网络字节顺序是TCP/IP中规定好的一种数据表示格式，可以保证数据在不同主机之间传输时能够被正确解释，网络字节顺序采用大端排序方式

9.TCP通信流程

服务器端

  1. 建立监听套接字
    • 监听：监听是否有客户端的连接
    • 套接字：是一个文件描述符，可以指向在内核中的一个缓冲区形成的伪文件
  2. 将监听套接字与服务器本地的端口号和IP地址绑定
  3. 设置监听，该监听套接字开始工作
  4. 阻塞等待，当有客户端发起连接，阻塞解除并接收客户端的连接，得到一个新的套接字，该套接字用于和该客户端通信
  5. 通信
  6. 通信结束，断开连接

客户端

  1. 创建一个用于通信的套接字
  2. 连接服务器，需要指定连接的服务器的IP与端口(指定了服务器的IP与端口，服务器端才能监听到)
  3. 连接成功，通信开始
  4. 通信结束，断开连接

11.TCP三次握手与四次挥手

TCP是一种面向连接的单播协议，在发送数据前需要在收发双方的内存里保存一份关于对方的信息，如IP地址，端口号等。TCP提供了一种可靠、面向连接、字节流、传输层的服务，采用三次握手建立一个连接，采用四次挥手关闭一个连接。

  • 16位端口号：告知主机报文段是来自那里，传给哪个应用程序或上层协议。TCP通信时，客户端通常使用系统自动选择的临时端口号
  • 32位序号(seq)：一次TCP通信过程中某一个传输方向上字节流的每个字节的编号。在某一方向的第一次发送时，该序号会被赋予一个初始值ISN，后续的报文段发送中，序号值起点seq = 前一次报文段总大小 + 初始序号值
  • 32位确认号(ack)：用作对另一方发送来的TCP报文段的响应。其值是收到的TCP报文段的序号值(seq) + 标志位的长度(SYN, FIN) + 数据长度
  • 六位标志位：
    • URG标志：表示紧急指针是否有效
    • ACK标志：表示确认号是否有效，携带ACK标志的TCP报文段叫做确认报文段
    • PSH标志：提示接收端应当立即从缓冲区读走数据
    • RST标志：重新连接的请求
    • SYN标志：请求建立连接
    • FIN标志：通知连接结束

三次握手

第一次握手：

  1. 客户端向发送标志位SYN=1的连接请求
  2. 生成一个随机的32位序号c_seq = J

第二次握手:

  1. 服务器端接收到请求，将ACK = 1
  2. 回复客户端一个确认号s_ack = c_seq + 数据长度 + 1(ACK的长度)
  3. 服务器端发送一个标志位SYN = 1的连接请求，生成一个32位随机序号s_seq = k

第三次握手：

  1. 客户端接收到请求后，将ACK = 1
  2. 回复服务器端一个确认号c_ack = s_seq + 数据长度 + 1

四次挥手

第一次挥手：

​ 客户端发送一个FIN = 1的断开连接请求报文段c_seq = his_seq + data

第二次挥手：

​ 服务器端接收到请求，回复一个ACK = 1的确认报文段，s_ack = c_seq + 1(除了数据之外还收到了FIN信号)

第三次挥手：

​ 服务器端也发送一个FIN = 1 的断开连接请求报文段s_seq = his_seq + data

第四次挥手：

​ 客户端收到请求，回复一个ACK = 1 的确认报文段c_ack = s_seq + 1

下一次通信发送报文段的序号头(seq)，总等于上一次通信结束后收到的ack值

滑动窗口

滑动窗口是为了解决发送端和接收端发送和接收的速率不同步导致的缓冲区溢出问题，用于改善网络通信中的吞吐量。TCP中采用滑动窗口来进行传输控制，滑动窗口的大小意味着接收方还有多大的缓冲区可以用于接收数据

TCP状态转换

建立连接时的状态转换：

  1. 客户端调用connect()之后发送SYN信号，此时客户端状态变为SYN_SENT；
  2. 服务器端接收到客户端的SYN信号，回复ACK同时发送SYN信号，服务器此时由LISTEN状态转换为SYN_RCVD状态；
  3. 客户端接受到服务器的SYN信号，转变为ESTABLISHED状态，同时发送ACK信号；服务器接收到客户端的ACK信号，状态转变为ESTABLISHED，至此，三次握手完成。

断开连接时的状态转换：

  1. 由主动发起断开请求的主动方发送FIN信号给被动方，主动方状态变为FIN_WAIT_1；

  2. 被动方接受到FIN信号之后进入CLOSE_WAIT状态，并发送ACK信号给被动方，但由于防止某些任务还没有完成，被动方可以不立即发送FIN给主动方；

  3. 当主动方接受到ACK但还没有接受到FIN时会进入FIN_WAIT_2状态，即半关闭状态，此时可以读取被动方的信息，但不能发送；

  4. 当主动方接受到被动方的FIN(被动方调用close())信号时，主动方发送一个ACK信号给被动端，自身进入持续2msl时间的TIME_WAIT状态，防止被动端没有接收到ACK

端口复用常见的用途：

  • 服务器突然关闭，绑定的端口未释放；
  • 程序突然退出，系统还未释放端口

12.IO多路转接技术

select

  1. 首先构造一个关于文件描述符的列表，将要监听的文件描述符添加到该列表中
  2. 调用一个系统函数，监听该列表中的文件描述符，直到这些描述符中的一个或者多个进行IO操作时，该函数才返回。
    • 这个函数是阻塞的
    • 函数对文件描述符的检测是由内核完成的
  3. 在返回时，它会告诉进程有多少(哪些)描述符要进行IO操作
  • 功能：建立一个文件描述符表，用于委托内核查询那些文件描述符进行了IO操作
  • 参数：
    • nfds：委托内核查询的最大的文件描述符 + 1
    • readfds：读集合，委托内核检测那些文件描述符的读的属性发生了变化
    • timeout：设置的超时时间，如果为NULL则永久阻塞

poll

读事件：

  • POLLIN：普通或优先带数据可读
  • POLLRDNORM：普通数据可读
  • POLLRDBAND：优先级带数据可读
  • POLLPRI：高级优先数据可读

写事件：

  • POLLOUT：普通或优先带数据可写
  • POLLWRNORM：普通数据可写
  • POLLLWRBAND：优先级带数据可写

错误事件：

  • POLLERR：发生错误
  • POLLHUP：发生挂起
  • POLLNVAL：不是打开的文件

poll相较于select的优势在于没有文件描述符数量的限制；共同的缺点在于包含大量文件描述符的数组需要在用户态和内核之间来回拷贝。

epoll

在内核中创建一个事件池，将需要检测的文件描述符传入该池，委托内核进行检测，内核将所有进行IO操作的文件描述符再放入一个双链表结构中传出

  • 功能：对epoll示例进行管理，增加需要检测的文件描述符，删除信息，修改检测功能等
  • 参数：
    • epfd：操作epoll示例的文件描述符
    • op需要进行的操作
      • EPOLL_CTL_ADD
      • EPOLL_CTL_MOD
      • EPOLL_CTL_DEL
    • fd：需要被检测的文件描述符
    • event：一个结构体，包含被检测文件需要检测的事件以及fd

  • 广播只能在局域网上使用
  • 服务器创建用于广播的套接字，向广播地址发送信息
  • 客户端绑定广播地址或INADDR_ANY与端口，接收信息

将服务器套接字设置为组播：

将客户端套接字加入多播组：

  • 服务器端创建用于组播的套接字，传入多播地址，将客户端默认指定为多播地址，向多播地址发送信息
  • 客户端创建一个套接字，绑定组播的IP或INADDR_ANY与组播端口
  • 将客户端套接字加入多播组

组播和广播时，服务器不绑定，客户端绑定
